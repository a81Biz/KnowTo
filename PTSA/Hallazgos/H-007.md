# H-007 — Estados de validación `rejected` y `aprobado_con_errores` no documentados en CLAUDE.md

## Descripción

CLAUDE.md documenta los estados de `validacion_estado` en `fase4_productos` como:
- `aprobado`
- `aprobado_por_fallback` 
- `revision_humana`

Los datos reales en BD muestran estados adicionales no documentados:
- `rejected` — Producto rechazado completamente por violaciones de dominio graves
- `aprobado_con_errores` — Producto aprobado pero con violaciones detectadas (no bloqueantes)

## Evidencia

Query psql en `fase4_productos` del proyecto "Curso Preparatorio CENEVAL":
```
P1 → rejected
P4 → aprobado_con_errores
P2, P3, P5, P6, P7, P8 → aprobado
```

CLAUDE.md solo menciona `'aprobado'`, `'aprobado_por_fallback'`, `'revision_humana'`.

## Impacto

Bajo para el código (los estados funcionan correctamente). Medio para mantenimiento: un desarrollador nuevo no sabrá qué estados existen ni qué significan. En particular, la diferencia entre `rejected`, `revision_humana` y `aprobado_con_errores` no es obvia y requiere leer el código fuente.

## Recomendación

Actualizar CLAUDE.md con la tabla completa de estados de `validacion_estado`:
| Estado | Significado |
|:---|:---|
| `aprobado` | Pasa todos los checks del EC0366RulesEngine |
| `aprobado_con_errores` | Aprobado con violaciones no bloqueantes detectadas |
| `aprobado_por_fallback` | Aprobado porque la validación falló con excepción (fallback safe) |
| `revision_humana` | Requiere revisión manual antes de usar |
| `rejected` | Rechazado por violaciones graves del dominio |
